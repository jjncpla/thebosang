# TBSS 신뢰도·자동화 우선순위 보고서 (2026-05-04)

> 텔레그램 1차 학습(internal_topic 5개 방, 1,000 메시지 + 첨부 메타데이터 383건) +
> 기존 OneDrive/Telegram 4,526개 양식 학습 결과를 결합한 *지금 무엇을 만들어야 신뢰도가 가장 빨리 올라가는가* 의 답.

---

## 결론 한 문장

**일정 동기화 → 직업력 자동 인입 → 진찰요구서 OCR 골드셋 → 공단 SMS 사건 매칭** 순으로 깔면 1주 안에 자동화 신뢰도가 측정 가능한 수준으로 올라간다. 텍스트 메시지만으로 이미 90% 이상 정형화돼 있어 OCR보다 *메시지 파서*가 더 빠른 ROI.

---

## 우선순위 1 — 특진 일정 동기화 캘린더 (즉시 구현, ROI 압도)

### 근거
- 일정방 200건 중 헤더 정형도 **96%** (`<{병원} {상병} 특진 일정[ 변경| 취소]>`)
- 일정 `1차/2차/3차 : YYYY.MM.DD.(요일) HH:MM` 패턴
- 현재 TBSS는 `HearingLossExam` 모델은 있지만 *입력은 100% 수동*
- 노무사들이 이미 텔레그램에 일정 올리는 습관이 강함 → 일정방 메시지를 단일 진실 소스로 쓸 수 있음

### 자동화 설계
1. **메시지 파서** (`lib/parsers/exam-schedule.ts`) — 헤더/회차 정규식 + 변경/취소 분기
2. **신규 페이지** `/calendar/exams` — 특진 일정 통합 캘린더 (FullCalendar 또는 자체)
3. **자동 인입 워커** (Phase 2 / 데몬 모드) — 일정방 신규 메시지 → `HearingLossExam` upsert
4. **임시 PoC**: 노무사가 메시지 텍스트 붙여넣기 → 즉시 파싱 결과 미리보기 + 사건 매칭 (자동 매칭 안 되면 검색 UI)

### 신뢰도 메트릭
- 헤더 매치율: 측정 완료 96%
- 회차/날짜 추출률: 정규식 + LLM 보정 후 98% 목표
- DB 사건 매칭률: 재해자명 + 상병 + 병원으로 fuzzy match — 다음 사이클에서 측정

### 예상 공수
- 파서: 0.5d
- 캘린더 페이지: 1d
- 사건 매칭 fuzzy 알고리즘: 0.5d
- **합계 2d**

## 우선순위 2 — 직업력 자동 초안 (폐질환·근골 ROI 즉시)

### 근거
- 폐질환·근골 접수 메시지 정형도:
  - `적용사업장 :` / `직종 :` / `직력 :` / `상병명 :` 4개 필드 동기화율 ~90%
  - 근골방 ICD 코드 `[Mxxxx]` 자동 추출 가능 (M751 회전근개 / M511 요추간판탈출 등)
- 폐질환은 **직업력이 처분(승인/불승인) 결과를 좌우** — 텔레그램에서 보이는 처분 비율: 불승인 32 / 승인 18 = 1.7배. 직업력 정밀도가 곧 신뢰도.
- 현재 TBSS는 `Case.workHistory` JSON 필드 있지만 입력 UI 미흡

### 자동화 설계
1. **텍스트 파서** (`lib/parsers/work-history.ts`) — 사업장/직종/직력 추출 + 기간 normalization (`약 22년 7개월` → `{years: 22, months: 7}`)
2. **직업력 입력 UI** 개선 — 메시지 붙여넣기 영역 → 자동 분해 → 행 추가 가능
3. **ICD → 부위 매핑 테이블** (`lib/constants/icd-musculoskeletal.ts`)
   - M170 양측슬관절 OA, M171 무릎 OA, M4806 요추척추증, M4802 흉추척추증, M511 요추간판탈출, M751 회전근개, M754 견관절증후군 등
   - 장해등급 페이지(`app/grade/`)와 연동
4. **`Case.workHistoryRaw`에 원본 메시지 텍스트 그대로 보존** (소스 추적·재파싱 가능)

### 신뢰도 메트릭
- 사업장+직종 동시 매치율: 텔레그램 기준 ~90%
- LLM 보정 후 95% 목표
- 정밀도(precision): 사람이 검수해서 수정한 행/전체 행

### 예상 공수
- 파서: 1d
- UI 개선: 1d
- ICD 테이블: 0.5d
- **합계 2.5d**

## 우선순위 3 — 진찰요구서 OCR 골드셋 + 회귀 테스트

### 근거
- 진찰요구서 방에서 **표준 양식 PDF 100건** 확보 (소음성 난청 100%, 텍스트 메시지 99% 정형)
- 기존 OCR 룰(`lib/ocr/parsers/`)이 진찰요구서를 얼마나 정확히 파싱하는지 *측정값이 없음*
- 100건 = 최소한의 회귀 테스트셋 (k-fold 가능)

### 자동화 설계
1. **다운로드 + 정리**: corpus/internal_topic/.../media/{msg_id}__{filename}.pdf → `tools/telegram-mcp/corpus/_gold/진찰요구서/`
2. **수동 라벨링** (또는 LLM 1차 라벨링 + 사람 검수): 재해자명·상병·진찰종류·병원·발급일·진찰요구사항
3. **회귀 테스트**: `lib/ocr/parsers/exam-request.ts` 기존 룰 vs 골드셋 비교
4. **신뢰도 등급 표시**: TBSS 양식 자동완성 UI에서 "OCR 신뢰도: 0.92" 같은 점수 노출

### 신뢰도 메트릭
- 필드별 정확도 (재해자명, 상병명, 발급일 등)
- precision/recall
- 회귀 테스트 통과율 (CI 통합 가능)

### 예상 공수
- 다운로드: 0.5d (lock 해결 후)
- 라벨링: 1d (LLM 1차 + 사람 검수 30분)
- 회귀 테스트 정비: 0.5d
- **합계 2d**

### 의존성
- 텔레그램 MCP 세션 lock 해결 필요 (현재 `database is locked`로 다운로드 막힘 — 4번 자동화에서 정기 fetch 시 같이 해결)

## 우선순위 4 — 공단 SMS 사건번호 자동 매칭

### 근거
- 중증&유족방에 forward된 공단 Web발신 SMS 31건, **사건번호 100% 추출**
- 사건번호(`2026-XXXXXXX`) = 공단 측 unique key, TBSS `Case.caseNo`와 1:1 매칭 가능
- 노무사 액션: 공단에서 받은 SMS를 텔레그램에 forward 한 번이면 사건 자동 동기화

### 자동화 설계
1. **SMS 파서** (`lib/parsers/kosha-sms.ts`) — 사건번호·심의일·질판위지역·상병 추출
2. **사건 매칭** — `Case.caseNo` 단순 lookup → 일치 시 status/qpwSchedule 자동 갱신
3. **신규 필드** (`Case.qpwScheduledAt`, `Case.qpwRegion`) — Phase 마이그레이션 필요
4. **미매칭 SMS 알림** — 사건번호가 TBSS에 없는 경우 inbox에 표시 (신규 사건 등록 유도)

### 신뢰도 메트릭
- 사건번호 추출률: 텔레그램 기준 100%
- TBSS 매칭률: TBSS 등록 사건 수 대비 (다음 사이클에서 측정)

### 예상 공수
- 파서: 0.5d
- 매칭 + UI: 1d
- **합계 1.5d**

## 우선순위 5 — 결정통지서·평균임금산정 자동완성 강화

### 근거
- 폐질환방·중증유족방에 결정통지서 PDF 다수 확인 (COPD 불승인 통지서 등)
- 기존 학습(`docs/forms-learning/결정통지서_OCR_정확도_비교.md`)에서 양식 구조 이해도 있음
- 텔레그램 첨부로 *최신 양식 변경* 추적 가능 (공단이 양식 갱신할 때마다 노무사가 채팅방에 올림)

### 자동화 설계
1. **양식 변경 모니터링** — 파일명 + 메시지 텍스트 차이 자동 감지 → 양식 spec 갱신 알람
2. **결정통지서 OCR 정확도 골드셋 확장** — 진찰요구서 100건과 같은 방식으로
3. **평균임금산정내역서 자동 인입** (이미 [progress](file:../progress_form_analysis.md)에 학습 완료) — 텔레그램에 신규 평임 내역서 올라오면 즉시 인입 후 `Case.parentCaseId` 자동 설정

### 예상 공수
- 양식 변경 감지: 1d
- 결정통지서 골드셋: 1d (진찰요구서와 동시 진행 시 +0.5d)
- 평균임금 자동 인입 강화: 1d
- **합계 3d**

## 우선순위 6 (백로그) — TF·노무사 단위 분석 페이지

### 근거
- 텔레그램에서 TF/담당자가 메시지 메타데이터로 일관 등장
- "TF별 처리 건수, 처분 비율, 평균 처리 일수" 같은 KPI 자동 계산 가능
- 권한 매트릭스상 *조직관리자* 페이지로 적절

### 자동화 설계
1. TF/담당자 단위 dashboard
2. 텔레그램 메시지 카운트 → 활동성 지표
3. 처분 결과 분포 (승인/불승인/이의제기/심사청구)

→ Phase 5 이후 검토.

## 본 보고서가 답하는 핵심 질문 (이정준 요청)

> "지금 개발하고 있는 페이지의 방향성을 명확히하고 싱크로나이징이나 자동서식 완성, 직업력 산정 등의 핵심 기능을 확실히하고 신뢰도를 높이기 위함"

- **싱크로나이징** → 우선순위 1(일정방)·4(공단 SMS)에서 텔레그램 메시지를 단일 진실 소스로 활용. 노무사가 따로 입력 안 해도 자동 인입.
- **자동서식 완성** → 우선순위 3(진찰요구서 골드셋)·5(결정통지서/평임). 진찰요구서 100건이 *지금 당장 측정 가능한* 회귀 테스트 인프라.
- **직업력 산정** → 우선순위 2. 메시지 정형도 90%, ICD 자동 추출, 폐질환 처분 결과(승인/불승인 = 18:32) 학습 가능 → 직업력 정밀도가 곧 처분 적중률.

핵심 기능 신뢰도는 **측정 가능한 메트릭으로 환산**해야 의미가 생긴다 — 위 5개 우선순위 모두 측정값이 정의되어 있으므로 매주 학습 보고서로 추세 추적 가능.
