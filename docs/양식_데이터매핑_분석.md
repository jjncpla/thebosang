### 3.2 핵심 양식 상세

#### 3.2.1 DISABILITY_CLAIM (장해급여청구서) — 425건 ★ 최다
**현재 TBSS 매핑**: HearingLossDetail.bankName/Account/AccountHolder/AccountType, complicationPart/Hospital
**폴더 사례에서 추출된 추가 필드**:
- **장해등급 산정 정보**: "기초산정 일반 11급05호 두 귀의 청력이 모두 1미터이상의 거리에서는 작은 말소리를 알아듣지..." → 장해등급 + 등급정의 + 신체부위 자동 표기 필요
- **의료기관 정보**: 의료기관명, 의사 성명, 의사면허번호 — 장해진단서 발급 의료기관
- **이송비 청구**: 산재관리의사 여부, 이송비용 — 별도 청구 항목

**갭 분석**:
- 현재 TBSS DISABILITY_CLAIM은 소음성 난청 전용으로 결합. **6개 미구현 상병별 장해급여청구서 신규 필요**
- 의료기관 정보 입력 폼이 없음 → Patient 모델 또는 별도 MedicalInstitution 테이블

#### 3.2.2 NOISE_WORK_CONFIRM (소음작업종사사실확인서) — 36건
**핵심 필드**:
- 사업장 취업 이전 거주지 (예: "포항시 죽도동")
- 성명 / 연락처 / 주소
- 사업개시번호
- 프레스, 자동선반 등 (작업 장비명)

**TBSS 현재 구조**:
- `Case.workHistory` JSON 필드에 기록
- 회사명, 부서, 작업내용, 소음노출 여부, 소음레벨

**갭**:
- "취업 이전 거주지", "고향 및 거주지" — TBSS workHistory에 없는 필드. 이는 노인성 난청 vs 소음성 난청 구별을 위한 거주환경 정보로 중요. **추가 필요**.

#### 3.2.3 DUST_WORK_CONFIRM (분진작업종사사실확인서) — 19건
**핵심 필드** (관련 데이터 통합):

### 4.2 우선순위 ★★★ 양식 상세

#### 4.2.1 WAGE_CORRECTION_CLAIM (평균임금정정청구서) — 42건 ★ 최우선

**실무 흐름**:
1. 공단이 평균임금산정내역서 (AVG_WAGE_REPORT) 발신
2. 노무법인이 검토 → 적용평균임금 < 통계임금 비교임금 등 부당 시
3. 평균임금정정청구서 작성 → 공단 제출
4. 결과 통지 (인용/기각)

**TBSS 현재 모델**:
- `WageReviewData` 모델 존재 (caseId, baseAvgWage, statWage*, comparisonWage, finalSelectedWage, claimDate, decisionResultDate 등)
- ObjectionCase.wageCorrectStatus (접수 대기/진행중/처분완료)
- **UI 폼 미확인 → 폼 구현 필요**
- **PDF 자동생성 미구현 → 신규 추가 필요**

**제안 필드** (사례 분석 기반):
- 사건 정보: 재해자명, 주민번호, 사업장명
- 수령희망 은행 및 계좌번호 (예금주, 은행명, 계좌번호)
- 근거법령: 근로기준법, 산업재해보상보험법 시행령
- 정정 청구 사유 (자유 텍스트, 비교임금 분석 결과 첨부)

#### 4.2.2 AVG_WAGE_REPORT (평균임금산정내역서) — 45건 ★ 자동 인입 핵심

**OCR 추출 필드 (실제 사례 기반)**:
- 회사명, 사업종류, 직종
- 채용일, 산정사유발생년월일
- 근로기준법 평균임금: 일당 × 통상근로계수(0.73) = 평균임금
- 산재보험법 특례임금: 노동통계 기반 365일 적용분기 / 월별일수·금액 / 특례임금액
- **부산질병판정위원회 / 서울질병판정위원회 결정일** — 평균임금 결정과 연관
- **증감률** (적용평균임금 옆) — 정정 청구 판단 직접 트리거

**자동 인입 매핑 → WageReviewData**:
| 양식 필드 | WageReviewData 매핑 |
|---|---|
| 회사명 | workplaceName |
| 적용평균임금 | appliedWage |
| 근기법 평균임금 (일당×0.73) | baseAvgWage |
| 산재보험법 특례임금 | comparisonWage |
| 직종 | occupation1 |
| 사업체노동력조사 | statWageBase, statWageRate, statWageFinal |
| 증감률 | changeRate |
| 결정일자 | decisionDate |

**정정청구 트리거 자동화**:
```
IF appliedWage < statWageFinal × 0.95 THEN
  WageReviewData.reviewResult = "정정청구 검토 필요"
  ObjectionCase.wageCorrectStatus = "접수 대기"
ELSE
  reviewResult = "정정청구 불요"
END
```

## 6. 평균임금 데이터 자동 인입 파이프라인

### 6.1 현재 한계
- 노무사가 평균임금산정내역서 PDF를 수동 검토
- WageReviewData 입력 폼이 없거나 미구현
- 정정청구 판단을 노무사 개인 경험에 의존

### 6.2 자동화 제안
```
1. 정보공개로 평균임금산정내역서 PDF 수신
   ↓
2. TBSS에 PDF 업로드 (Case 첨부)
   ↓
3. OCR (Tesseract + tessdata_best, kor+eng)
   ↓
4. 정규식 + 키워드 기반 필드 추출:
   - 회사명, 직종
   - 일당, 통상근로계수, 평균임금
   - 사업체노동력조사 적용분기
   - 월별 일수/금액 → 365일 합계
   - 특례임금액
   ↓
5. WageReviewData 자동 INSERT/UPDATE
   ↓
6. 트리거 룰:
   - appliedWage < statWageFinal × 0.95 → reviewResult = "정정청구 검토"
   - 자동으로 ObjectionCase.wageCorrectStatus 업데이트
   ↓
7. 노무사 검토 → 평균임금정정청구서 PDF 자동생성
```

### 6.3 OCR 정확도 검증 결과
- 폴더 45건 평균임금산정내역서 → 100% 텍스트 확보
- "최초 평균임금산정내역", "관리번호", "회사명", "성명", "일당", "통상근로계수", "평균임금" 등 핵심 라벨 OCR 인식 양호
- 다만 인쇄 흐림·필기 메모 등은 부분 인식 — 노무사 최종 검토 필수

   - PDF 업로드 → OCR → 구조화 추출 → DB 인입

### 8.2 중기 작업 (1-3개월)

   - "취업 이전 거주지" 필드 추가 (노인성 vs 소음성 구별 자료)

7. **정보공개 자료 묶음 자동 분리**
