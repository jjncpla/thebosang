## 2. 개발 원칙 (반드시 준수)

1. **수술적 수정**: 기존 UI·로직 건드리지 않고 필요한 부분만 변경
2. **전체 재작성 금지**: 명시적 요청 없으면 기존 파일 전체 재작성 안 함
3. **기존 스타일 유지**: 인라인 스타일 / Tailwind 혼재 현황 그대로 존중
4. **TS 상수로 데이터 관리**: 정적 데이터는 동적 로딩 대신 TS 상수 하드코딩
5. **긴 지시사항은 .txt 파일로**: Claude Code CLI 터미널 붙여넣기 한계 대응
6. **임의 패키지 설치 금지**: 사전 협의 없이 새 패키지 추가 금지

## 3. 사건관리 PRD (핵심 스키마)

### 3-1. 핵심 설계 원칙

- **Patient(재해자)** 가 최상위 엔티티. 1명이 복수 상병(사건) 가능
- **Case(사건)** 은 항상 특정 상병에 귀속. 상병이 다르면 별개 사건
- **파생 사건**: 평균임금 정정 청구 시 원본 `caseId`를 `parentCaseId`로 연결
- **B방식 FK**: `Case → ObjectionReview → ObjectionCase` 명시적 `caseId` FK

### 3-2. 상병 계열

- **기획사건**: 소음성 난청 / COPD / 진폐증
- **일반산재**: 근골격계 / 업무상 사고 / 직업성 암 / 유족 / 기타

### 3-3. CaseStatus Enum (기획사건용)

```
CONSULTING              상담 중 (초진 전, 약정 전)
CONTRACTED              약정 완료 / 진행 개시
DOC_COLLECTING          기본자료 수집 중
SUBMITTED               공단 접수 완료 (우편 발송)
EXAM_REQUESTED          특진진찰요구서 수령
EXAM_CLINIC_SELECTED    특진병원 선택확인서 제출 완료
EXAM_SCHEDULED          특진 일정 확정
IN_EXAM                 특진 진행 중
EXAM_DONE               특진 완료 (3차 종료 + 의무기록지 수령)
EXPERT_REQUESTED        전문조사진찰요구서 수령
EXPERT_CLINIC_SELECTED  전문조사기관 선택 완료
EXPERT_DONE             전문조사 완료
BANK_REQUESTED          통장사본 요청 수령 (전화)
BANK_SUBMITTED          통장사본 제출 완료
DECISION_RECEIVED       장해등급결정통지서 수령
REVIEWING               노무사 검토 중 (평정/적사변 실익)
INFO_REQUESTED          정보공개청구 중
APPROVED                승인 확정
REJECTED                불승인
CLOSED                  종결
OBJECTION               이의제기 진행 중 (이의제기 관리로 이관)
WAGE_CORRECTION         평균임금 정정 청구 (파생 사건 생성)
```

### 3-4. Case 공통 필드

```prisma
caseType        Enum  (HEARING_LOSS / COPD / PNEUMOCONIOSIS / MUSCULOSKELETAL /
                       OCCUPATIONAL_ACCIDENT / OCCUPATIONAL_CANCER / BEREAVED / OTHER)
parentCaseId    String?   // 파생 사건 연결 (평균임금 정정 청구 등)
salesManagerId  String    // User FK (현재 String에서 변경)
caseManagerId   String    // User FK
branchManagerId String    // User FK
```

### 3-5. 소음성 난청 절차 흐름

```
CONSULTING
  └─ 초진 통과 (편측 PTA 40dB↑) + 약정서·위임장 수령 → CONTRACTED
  └─ 기본자료 수집 시작 → DOC_COLLECTING
  └─ 청구서식 + 우편 발송 (스캔 후 텔레그램 난청 공유방 공유) → SUBMITTED
  └─ 특진진찰요구서 수령 (웹팩스) → EXAM_REQUESTED
  └─ 특진병원 선택확인서 제출 → EXAM_CLINIC_SELECTED
  └─ 특진 일정 확정 → EXAM_SCHEDULED
  └─ 1차 특진 → IN_EXAM (examRound=1)
  └─ 2차 특진 → IN_EXAM (examRound=2)
  └─ 3차 특진 + 의무기록지 발급 → EXAM_DONE
  └─ 전문조사진찰요구서 수령 → EXPERT_REQUESTED
  └─ 전문조사기관 선택 → EXPERT_CLINIC_SELECTED
  └─ 전문조사 실시 → EXPERT_DONE
  └─ 통장사본 요청 수령 → BANK_REQUESTED
  └─ 통장사본 제출 → BANK_SUBMITTED
  └─ 장해등급결정통지서 수령 → DECISION_RECEIVED
  └─ 노무사 검토 → REVIEWING
      ├─ 정공 필요 → INFO_REQUESTED → 자료 수령 후 재검토
      ├─ 이상 없음 → CLOSED
      ├─ 평균임금 정정 → WAGE_CORRECTION (파생 사건)
      └─ 불승인 → REJECTED
          ├─ 종결 → CLOSED
          └─ 이의제기 → OBJECTION
```

## 14. 상병별 상세페이지 자동연계 PRD 템플릿

> **배경**: COPD 야간 QA 작업으로 자동연계 9개 메커니즘 완성. 다른 상병은 도메인 차이가 커서 무작정 복붙 금지. 각 상병 상세페이지 작업 시 본 템플릿을 작성한 후 구현 착수.
> **참조 메모리**: `qa_other_diseases_pattern_apply.md`, `qa_copd_overnight_summary.md`, `qa_copd_decisions_prd.md`, `d8_status_enum_unify_plan.md`
> **상세 체크리스트**: `docs/기획서_기획.md` §5-6 참조

### 14-1. PRD 작성 의무 섹션 (5개 필수)

각 상병별 상세페이지 작업 PRD에 다음 5개 섹션을 **반드시 포함**:

#### 1) Schema 분석
- 1:1 vs 1:N 회차 모델 여부
- 처분 관련 필드 목록 (`disposalType`, `disposalDate`, `approvalDate`, `disabilityApprovalDate` 등)
- 통지수령일 컬럼 유무 — 없으면 처분일 fallback 사용 가능성 검토
- 일정 필드 (캘린더 동기화 대상): `expertScheduleDate`, `committeeReviewDate`, `precisionExamDate` 등
- status 도메인: 영문 enum / 한글 / GENERAL_STATUS 중 어느 것

#### 2) 처분 흐름
- 단일 처분 vs **요양/장해 2단계 처분 분리** (MUSCULO/OCC_ACC)
- 회차 개념 유무 (PNEUMO는 COPD와 다른 회차 구조)
- 이의제기 대상 status (`승인`/`불승인`/`요양승인`/`장해승인` 등 — 정책 결정 필수)

#### 3) 자동연계 적용 범위 (COPD 9개 메커니즘 중 선별)
| # | 메커니즘 | 적용 여부 | 비고 |
|---|----------|----------|------|
| B1 | POST /api/cases 시 Detail+회차 자동 생성 | (1:1은 보류 권장) | UI 첫 저장 시 upsert로 충분 |
| B2 | `syncFrom[X]Decision` (처분 → ObjectionReview upsert) | 필수 검토 | **신규 함수 격리** |
| B3 | autoFill GET 한글 status OR 확장 (`/api/objection/review`) | 필수 검토 | status 추가 명단 사용자 컨펌 |
| B4 | autoFill GET → `/api/objection/cases` 동일 OR 확장 | 필수 검토 | B3와 동기 |
| B5 | examClaimDeadline = 기준일 +90일 | 필수 검토 | 기준일 정의 (처분일/수령일) |
| B6 | `derive[X]Status` (회차 → Case.status 도출) | 1:1은 단순 함수 | STATUS_PRIORITY reduce 불필요 |
| B7 | `sync[X]CaseEvents` (캘린더 멱등 동기화) | 일정 필드 있을 때만 | memo 표식 `[X_AUTO_*]` |
| B8 | 캘린더 `[X]` prefix 라벨 | (UI 1줄) | diseaseType 컬럼 사용 |
| B9 | Patient DELETE cascade | (이미 caseType-agnostic) | 별도 작업 불필요 |

#### 4) 회귀 위험 평가
- HL/COPD 자동연계에 영향 없는지 명시 (caseType 분기 격리 원칙)
- 공용 함수(`syncFromObjectionReview` / `syncFromCaseStatus`)에 caseType 분기 누적 금지 — 신규 함수 신설 우선
- 기존 가드 패턴 재사용 (`lib/case-sync.ts:72-77` "비-HL 케이스 closedReason 가드" 참조)

#### 5) 마이그레이션 필요성
- 통지수령일 컬럼 신규 추가 여부 (`disposalNoticeReceivedAt DateTime?`)
- D8 status enum 통합 영향 (한글 status → 영문 enum 전환 시점)
- 기존 사건 데이터 backfill 필요 여부

### 14-2. 우선순위

| 상병 | 우선순위 | 작업 묶음 |

### 14-3. 신규 sync 함수 명명 규약

- 처분 → ObjectionReview 동기화: `syncFrom[X]Decision` (예: `syncFromMusculoDecision`)

- 캘린더 memo 표식: `[MUSCULO_AUTO_*]` / `[OCCACC_AUTO_*]` / `[PNEUMO_AUTO_*]` 등 (멱등 동기화용)
