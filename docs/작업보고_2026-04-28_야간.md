## 1. /cases 페이지 성능 최적화

### 진단 (Before)
- **클라이언트 컴포넌트** (`"use client"`) → 마운트 후 `fetch` → 흐린 스켈레톤 6초~ 표시
- `/api/cases` API: **인덱스 0개**, 페이지네이션 없음, `hearingLoss` 100+ 필드 전부 select
- `loading.tsx` 부재 → 첫 페인트 시 빈 상태

### 적용 (After)
| 영역 | 변경 |
|------|------|
| **DB 인덱스** | `Case` 모델에 11개 인덱스 추가 (`caseType+createdAt`, `status`, `tfName`, `branch`, `salesManagerId`, `caseManagerId`, `kwcOfficeName`, `kwcOfficerName`, `receptionDate`, `contractDate`, `patientId+caseType`) |
| **API select 슬림화** | `hearingLoss` 100+ 필드 → 5개 필드만 (`firstClinic`, `specialClinic`, `decisionType`, `disabilityGrade`). 다른 Detail 모델은 존재 여부만 |
| **페이지네이션** | `?paginate=true&limit=200&cursor=ID` cursor-based. 응답: `{ items, nextCursor, total? }` |
| **하위호환** | `paginate` 파라미터 없으면 기존 배열 응답 유지 (AppShell, objection/deadline 호환) |
| **stale-while-revalidate** | localStorage 캐싱 → 재방문 시 **흐린 화면 없이 즉시 데이터 표시** + 백그라운드 갱신 ("갱신 중..." 표시) |
| **Server-Timing 헤더** | `db;dur=Nms` 응답 헤더 — 브라우저 DevTools에서 DB 쿼리 시간 측정 가능 |
| **버그 fix** | "처분결과" / "장해등급" 컬럼이 잘못된 필드명(`disposalType`/`grade`)으로 항상 `-` 표시되던 버그 → `decisionType` / `disabilityGrade`로 정정 |

### 예상 효과
| 시나리오 | Before | After (예상) |
|----------|--------|--------------|
| 첫 방문 (캐시 없음) | 흐린 스켈레톤 1~3초 → 데이터 | 스켈레톤 0.3~0.8초 → 데이터 (인덱스+슬림화) |
| **재방문 (캐시 있음)** | 매번 흐린 스켈레톤 1~3초 | **즉시 표시** + 우상단 "갱신 중..." 후 fresh |
| 1만 건 사건 환경 | 전체 fetch (느림) | 200건씩 페이지네이션 (빠름) + 더 불러오기 |
| 필터 변경 | 매번 전체 재fetch | 필터별 별도 캐시 → 같은 필터 재선택 시 즉시 |

> **주의**: 운영 DB의 실제 데이터 규모를 모르므로 정확한 수치는 배포 후 Server-Timing 헤더로 측정 필요. 캐싱은 사용자 device의 localStorage(보통 5MB 한도) 내 작동.

## 2. /law 페이지 — 별표·첨부 기능 추가 (신규)

### 신규 모델
```prisma
model LawAttachment {
  id, lawId, attachmentType (별표/별지/첨부),
  number, title, description,
  fileName, contentType, fileSize, fileData (BYTEA),
  uploadedBy, createdAt, updatedAt
  @@index([lawId])
  @@index([lawId, attachmentType])
}
```
- 5MB 제한 (PG BYTEA 부담 최소화)
- PDF / PNG / JPEG 허용

### 신규 API
- `GET /api/law/attachments?lawId=&type=` — 메타데이터 목록 (fileData 제외)
- `POST /api/law/attachments` — multipart 업로드
- `GET /api/law/attachments/[id]/file?inline=1` — 바이너리 응답
- `PATCH /api/law/attachments/[id]` — 메타데이터 수정
- `DELETE /api/law/attachments/[id]`

### 신규 UI
- /law 페이지 새 탭: **별표·첨부**
- 좌측: 법령 그룹 선택 (LAW_GROUPS 12개)
- 우측: 첨부 목록 (별표/별지/첨부 카테고리별 그룹) + PDF iframe 미리보기 + 인쇄/다운로드 버튼
- **인쇄**: iframe `contentWindow.print()` (브라우저 PDF 뷰어 사용)
- 업로드 모달: 유형/번호/제목/설명/파일 입력

### LAW 검색 추가
- LawTab 사이드바 상단 "법령 검색…" 입력 → 그룹명/법령명 부분 일치 필터, 검색 시 모든 그룹 자동 펼침

## 5. 더미 데이터 시드 / 측정 — **SKIP** (사유 보고)

### 사유
- `.env`의 `DATABASE_URL`이 Railway 운영 DB를 가리키며 `.env` 직접 읽기 권한 차단됨
- 사용자 인스트럭션: "기존 실제 입력된 db는 건드리지 말 것"
- 운영 DB에 더미 데이터 (Patient 500/Case 1000) 추가 시 PII(주민번호) 컬럼에 가짜 데이터 섞임 → 실무 검색에 노이즈 발생
- BEGIN/INSERT/MEASURE/ROLLBACK 트랜잭션 측정도 운영 DB I/O 부하 + 실수 risk

### 대안 권장 (다음 세션 시 사용자 도움 필요)
- **로컬 PostgreSQL** 띄우기 (Docker 또는 Postgres.app 등)
- `.env.local`에 `DATABASE_URL=postgresql://localhost:5432/tbss_test`
- 더미 시드 → `prisma db push` → 인덱스/페이지네이션 효과를 EXPLAIN ANALYZE로 측정

### 운영 환경에서의 정성 측정
- 배포 후 `/cases` 첫 방문 시 DevTools Network 탭 → `/api/cases?paginate=true&limit=200&count=true` 응답의 `Server-Timing` 헤더 확인
- 사용자 체감으로 검증 가능 (재방문 시 즉시 표시 / 갱신 중... 표시 → 데이터 갱신)

## 6. 법령정보센터 OPEN API 안내 (자동 적용 불가, 사용자 결정 필요)

### 정보
- **API 제공 사이트**: <https://open.law.go.kr/>
- **신청 후 1~2일 내 승인** (사용자 직접 신청 필요)
- **공공데이터포털 경유**: <https://www.data.go.kr/data/15000115/openapi.do>
  - 개발계정: 10,000 트래픽/일
  - 운영계정: 활용사례 등록 시 트래픽 증액 가능
- 엔드포인트 예: `https://www.law.go.kr/DRF/lawService.do?OC=...&target=law&MST=법령ID&type=JSON`

### 권장 단계 (사용자 결정 후 별도 작업)
1. 사용자가 OPEN API 신청 → 승인 키 수령
2. `.env`에 `LAW_API_OC=발급받은ID` 추가
3. 현재 iframe 임베드(LawTab) → API 직접 호출 방식으로 마이그레이션 (선택 사항)
4. 별표/별지 자동 인입 — `/api/lawAttch.do?MST=&type=JSON` 호출 → DB 자동 채우기

> 현재 구현은 iframe 그대로 유지 + 사용자가 직접 별표 PDF 업로드. API 도입 시 "1회 자동 동기화" 작업 권장.
