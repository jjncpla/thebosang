## 2. 통합 캘린더 webhook 백업/redundancy 시스템 설계

### 현재 상태
- Telegram webhook + Calendar webhook으로 일정 수립
- 신뢰도 우려: 중복 일정 / 누락 일정 가능성

### 권고 아키텍처

#### A. Idempotency (중복 방지)
- 모든 webhook 처리 시 `(source, externalId)` 기준 unique key
- 이미 처리된 외부 이벤트는 skip + log

```prisma
model WebhookEvent {
  id           String   @id @default(cuid())
  source       String   // "telegram" | "google_calendar" | ...
  externalId   String   // 외부 시스템의 고유 ID (message_id 등)
  payload      Json
  processedAt  DateTime?
  createdAt    DateTime @default(now())
  @@unique([source, externalId])
  @@index([source, processedAt])
}
```

#### B. Redundancy (누락 방지)
**1차 라인**: webhook (real-time)
**2차 라인**: 1시간마다 cron (또는 Railway scheduled task)
- Telegram: `getUpdates`로 최근 24시간 메시지 재폴링 → WebhookEvent에 없는 것만 처리
- Calendar: `events.list` (updated >= 1h ago) → 동일 idempotency 검사

#### C. Rollback 시스템
- `CaseEvent` / `TimelineEvent` 모델에 `sourceEventId` (WebhookEvent FK) 필드
- 잘못된 webhook 처리 결과 발견 시 → `sourceEventId`로 모든 영향 row 추적 → 일괄 롤백

#### D. 모니터링
- Daily summary email/telegram: "어제 webhook 처리 N건, 실패 N건, 1시간 cron 보충 N건"
- Anomaly: 평소 평균 ±3σ 벗어나면 알림

### 구현 우선순위
1. **즉시**: WebhookEvent 모델 + idempotency 적용 (1일)
2. **단기**: 1시간 cron 보충 (Railway scheduled task) (3일)
3. **중기**: rollback 추적 + 모니터링 dashboard (1주)

### ⚠️ 주의
운영 시스템 변경 = 신중. 단계별 staged rollout 필요. 본 문서는 설계만 — 실 코드 변경은 사용자 승인 후 별도 PR.

2. **장의비 청구서** (`FUNERAL_BENEFIT`, 4건) — 유족급여와 함께
3. **간병급여 청구서** (`CARE_BENEFIT`, 1건) — 중증 장해자
4. **정보공개 결정통지서 OCR 인입** (`INFO_DISCLOSURE_DECISION`, 1건) — 공단 발신
5. **정보공개 자료 묶음 분리** (`INFO_DISCLOSURE_BUNDLE`, 22건) — 자동 분리 후 인입

### 우선순위
- ★★★ 즉시: INFO_DISCLOSURE_DECISION (이미 결정통지서 파서 패턴 활용 가능)
- ★★ 단기: INJURY_PENSION_CLAIM, FUNERAL_BENEFIT (양식 학습 후 좌표 매핑 또는 텍스트 PDF)
- ★ 추후: CARE_BENEFIT, INFO_DISCLOSURE_BUNDLE

### 학습 완료 후 작업 절차
1. 학습 종료 (~3일 후)
2. classify_v2 + extract_form_fields + bundle_for_review 재실행
3. 신규 양식 종류 + 빈도 확정
4. 빈도 5건 이상 양식 우선 자동생성 구현 (좌표 기반 PDF 또는 텍스트 PDF)
