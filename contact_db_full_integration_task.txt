## 세션 타입: 기능 개발 세션
## 작업 개요
임직원 인사카드(Contact DB)를 아직 연동되지 않은 전체 페이지의 담당자 필드에 연결한다.
ContactSelector / BranchSelector 공유 컴포넌트를 활용하여 텍스트 입력 또는 단순 User select를 교체한다.

---

## 수술적 수정 원칙
- 기존 파일 전체 재작성 금지 — 해당 input/select 요소만 교체
- 기존 저장된 데이터(문자열) 호환성 유지 — Contact.id FK가 아닌 이름 문자열 그대로 저장
- 기존 API 응답 구조 변경 최소화

---

## 작업 1: 상담관리 (/consultation)

대상 파일: `app/consultation/page.tsx`

### 1-1. 담당자 필드
현재: User 목록 select (users.map으로 렌더링) 또는 텍스트 입력
변경: ContactSelector로 교체

```tsx
import ContactSelector from '@/components/ui/ContactSelector'
// 또는 Step 0에서 확인한 실제 경로

// 기존 User select 또는 텍스트 input을 찾아서 아래로 교체:
<ContactSelector
  value={formData.managerName || ''}
  onChange={(name, mobile) => setFormData(p => ({ ...p, managerName: name }))}
  placeholder="담당자 이름 검색"
  firmType="TBOSANG"
/>
```

**주의**: 실제 필드명은 Step 0에서 확인한 것으로 교체. `managerName`이 아닌 `userId`를 저장하는 구조면, 
ContactSelector onChange에서 선택된 Contact의 userId를 같이 세팅:
```tsx
onChange={(name, mobile) => {
  // Contact 전체 데이터가 필요하면 ContactSelector를 확장하거나, name으로 조회
  setFormData(p => ({ ...p, managerName: name }))
}}
```

## 작업 5: 사건조회 (/inquiry) — 담당자 표시 보강

대상 파일: `app/inquiry/page.tsx`

### 5-1. 검색 결과 담당자 표시
현재: salesManagerId/caseManagerId → User.name 표시
보강: 담당자 옆에 연락처(Contact.mobile) 표시 추가

API(`app/api/inquiry/route.ts`)에서 담당자 조회 시 Contact도 같이 조회:
```typescript
// 기존 User.name 표시하는 부분에 Contact.mobile도 같이 반환
// Contact에서 name으로 조회하여 mobile 포함
```

이 작업은 **표시(display) 보강**이므로, API 응답에 mobile 필드 추가 + UI에 전화번호 표시만 하면 됨.
기존 검색 로직은 건드리지 않는다.

## 주의사항

1. **ContactSelector import 경로**: Step 0에서 확인한 실제 경로 사용 (예: `@/components/ui/ContactSelector` 또는 다른 경로)
2. **필드명 정확히 확인**: 각 페이지마다 담당자 필드명이 다름 (managerName, reviewManagerName, staffName 등). Step 0 grep 결과로 정확한 필드명 파악 후 교체.
3. **User FK 저장 구조인 경우**: 일부 페이지는 userId(FK)를 저장하는 구조일 수 있음. 이 경우 ContactSelector 선택 → Contact.userId → User FK로 저장하는 로직 필요. Contact에 userId가 없으면 이름 문자열로 저장.
4. **기존 데이터 호환**: 이미 저장된 텍스트 데이터는 그대로 유지. ContactSelector는 초기값으로 기존 문자열을 표시하므로 호환됨.
5. **파일 없으면 스킵**: Step 0에서 파일 존재하지 않는 페이지는 건너뛴다. 강제 생성하지 않는다.
