## 작업 개요

관리자 페이지 데이터 임포트 섹션에 기존 TF 데이터 삭제와 동일한 방식으로
아래 3가지 삭제 기능을 추가한다.

추가할 삭제 기능:
1. 상담내역 전체 삭제 (Consultation 테이블)
2. 처분검토 전체 삭제 (ObjectionReview 테이블)
3. 기일관리 전체 삭제 (ObjectionCase 테이블)

---

## 1단계 — 현재 코드 구조 파악

다음 파일들을 읽어서 기존 TF 삭제 구현 패턴을 파악한다:

1. 관리자 페이지 UI 파일 (아래 중 존재하는 것):
   - `app/admin/page.tsx`
   - `app/(dashboard)/admin/page.tsx`
   - `app/admin/import/page.tsx`
   - `app/(dashboard)/admin/import/page.tsx`

2. 기존 TF 삭제 API 라우트 (아래 중 존재하는 것):
   - `app/api/admin/delete-tf/route.ts`
   - `app/api/admin/tf/route.ts`
   - `app/api/tf/route.ts` (DELETE 메서드)

3. Prisma 스키마:
   - `prisma/schema.prisma`

위 파일들을 읽은 후 아래 작업을 진행한다.

## 2단계 — API 라우트 생성

기존 TF 삭제 API와 동일한 패턴으로 아래 3개 API 라우트를 생성한다.

### 파일 1: `app/api/admin/delete-consultations/route.ts`

```typescript
import { NextResponse } from 'next/server'
import { prisma } from '@/lib/prisma'
import { auth } from '@/auth'

export async function DELETE() {
  const session = await auth()
  
  if (!session || session.user?.role !== 'ADMIN') {
    return NextResponse.json({ error: '권한이 없습니다.' }, { status: 403 })
  }

  try {
    const result = await prisma.consultation.deleteMany({})
    return NextResponse.json({ 
      success: true, 
      deleted: result.count,
      message: `상담내역 ${result.count}건이 삭제되었습니다.`
    })
  } catch (error) {
    console.error('상담내역 삭제 오류:', error)
    return NextResponse.json({ error: '삭제 중 오류가 발생했습니다.' }, { status: 500 })
  }
}
```

> ⚠️ 참고: Prisma 모델명이 `Consultation`이 아닐 수 있다. schema.prisma를 확인하여 실제 모델명으로 수정한다.

### 파일 3: `app/api/admin/delete-objection-cases/route.ts`

```typescript
import { NextResponse } from 'next/server'
import { prisma } from '@/lib/prisma'
import { auth } from '@/auth'

export async function DELETE() {
  const session = await auth()
  
  if (!session || session.user?.role !== 'ADMIN') {
    return NextResponse.json({ error: '권한이 없습니다.' }, { status: 403 })
  }

  try {
    // ObjectionCase가 ObjectionReview에 FK로 연결된 경우 순서 주의
    // B방식: ObjectionCase → ObjectionReview 순으로 삭제
    const result = await prisma.objectionCase.deleteMany({})
    return NextResponse.json({ 
      success: true, 
      deleted: result.count,
      message: `기일관리 ${result.count}건이 삭제되었습니다.`
    })
  } catch (error) {
    console.error('기일관리 삭제 오류:', error)
    return NextResponse.json({ error: '삭제 중 오류가 발생했습니다.' }, { status: 500 })
  }
}
```

> ⚠️ schema.prisma에서 기일관리 관련 모델명을 확인하여 `objectionCase` 부분을 실제 모델명으로 수정한다.
> ⚠️ FK 제약 조건이 있는 경우, 자식 테이블부터 먼저 삭제해야 한다. 예: ObjectionCase가 ObjectionReview를 참조하면 ObjectionCase 먼저 삭제.

## 3단계 — 관리자 페이지 UI 수정

관리자 페이지의 데이터 임포트 섹션에서 기존 TF 삭제 카드를 찾아, 동일한 스타일로 아래 3개 카드를 추가한다.

### 추가할 UI 요소 (기존 TF 삭제 카드 패턴 그대로 복제 후 수정)

각 삭제 카드는 다음 정보를 가진다:

**상담내역 삭제 카드**
- 제목: "상담내역 전체 삭제"
- 설명: "상담 관리 페이지의 모든 상담내역 데이터를 삭제합니다."
- 버튼: "상담내역 삭제"
- API: DELETE /api/admin/delete-consultations
- 확인 메시지: "상담내역 데이터를 모두 삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다."

**처분검토 삭제 카드**
- 제목: "처분검토 전체 삭제"
- 설명: "처분검토 페이지의 모든 데이터를 삭제합니다."
- 버튼: "처분검토 삭제"
- API: DELETE /api/admin/delete-objection-reviews
- 확인 메시지: "처분검토 데이터를 모두 삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다."

**기일관리 삭제 카드**
- 제목: "기일관리 전체 삭제"
- 설명: "이의제기 기일관리 페이지의 모든 데이터를 삭제합니다."
- 버튼: "기일관리 삭제"
- API: DELETE /api/admin/delete-objection-cases
- 확인 메시지: "기일관리 데이터를 모두 삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다."

## 4단계 — 주의사항

1. **수술적 수정 원칙**: 기존 코드(TF 삭제 등)는 건드리지 않고, 신규 카드만 추가한다.
2. **Prisma 모델명 확인 필수**: schema.prisma에서 실제 모델명을 확인 후 API 코드에 반영한다.
3. **FK 순서**: ObjectionReview와 ObjectionCase 사이에 FK가 있으면, 자식(참조하는 쪽)을 먼저 삭제한다.
4. **prisma import 경로**: 기존 TF 삭제 API에서 사용하는 import 경로(`@/lib/prisma` 또는 `@/lib/db` 등)를 그대로 따른다.
5. **auth import 경로**: 기존 API에서 사용하는 경로를 그대로 따른다.
6. **로딩 상태**: 삭제 버튼 클릭 시 로딩 표시는 기존 TF 삭제와 동일한 방식으로 처리한다.
