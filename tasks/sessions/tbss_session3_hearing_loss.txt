## TASK 1: schema.prisma — HearingLossDetail / HearingLossExam 추가

파일: prisma/schema.prisma 끝에 추가. 기존 내용 건드리지 말 것.

```prisma
// ─────────────────────────────────────────────
// 소음성 난청 상세 정보
// ─────────────────────────────────────────────
model HearingLossDetail {
  id        String @id @default(cuid())
  caseId    String @unique
  // Case와 1:1 관계 (Case 모델에 역방향 관계 추가 필요)

  // 초진
  firstClinic            String?
  firstExamDate          DateTime?
  firstExamRight         Float?
  firstExamLeft          Float?
  firstExamSpeech        Float?
  passedInitialCriteria  Boolean  @default(false)
  isDisabilityRegistered Boolean  @default(false)
  isBig3                 Boolean  @default(false)

  // 직업력
  workHistory          Json?    // 배열 구조 (아래 참조)
  workHistoryMemo      String?
  lastNoiseWorkEndDate DateTime?

  // 접수/청구
  claimSubmittedAt  DateTime?
  claimNasPath      String?
  telegramSharedAt  DateTime?

  // 특진진찰요구서
  examRequestReceivedAt DateTime?
  examPeriodStart       DateTime?
  examPeriodEnd         DateTime?

  // 특진병원 선택
  specialClinic                   String?
  examClinicSelectionSubmittedAt  DateTime?

  // 전문조사
  expertRequestReceivedAt            DateTime?
  expertClinicOptions                Json?
  expertClinic                       String?
  expertClinicSelectionSubmittedAt   DateTime?
  expertDate                         DateTime?
  expertAttendedStaffId              String?
  expertMemo                         String?

  // 통장사본
  bankAccountRequestedAt  DateTime?
  bankAccountSubmittedAt  DateTime?

  // 결정
  decisionType      String?   // APPROVED / REJECTED
  decisionReceivedAt DateTime?
  approvedDisease   String?
  disabilityGrade   String?
  disabilityStatus  String?
  baseAssessment    String?
  finalAssessment   String?
  lumpSumAmount     Int?
  avgWage           Int?
  compensationPaidAt DateTime?

  // 검토/정공
  wageReviewMemo              String?
  adaptedWorkplaceReviewMemo  String?
  infoDisclosureRequestedAt   DateTime?
  infoDisclosureReceivedAt    DateTime?
  rejectionReason             String?
  reviewMemo                  String?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  exams HearingLossExam[]
}

// ─────────────────────────────────────────────
// 소음성 난청 특진 회차별 데이터
// ─────────────────────────────────────────────
model HearingLossExam {
  id                  String  @id @default(cuid())
  hearingLossDetailId String
  detail              HearingLossDetail @relation(fields: [hearingLossDetailId], references: [id], onDelete: Cascade)

  examSet   String  // INITIAL / RE / RE2
  examRound Int     // 1 / 2 / 3

  examDate            DateTime?
  attendedStaffId     String?
  airRight            Float?
  airLeft             Float?
  boneRight           Float?
  boneLeft            Float?
  speechRight         Float?
  speechLeft          Float?
  abrRight            Float?
  abrLeft             Float?
  impedanceRight      String?
  impedanceLeft       String?
  isReliable          Boolean?
  medicalRecordObtained Boolean @default(false)
  predictedGrade      String?
  memo                String?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([hearingLossDetailId, examSet, examRound])
}
```

Case 모델에 역방향 관계 추가 (수술적으로 Case 모델 내부에 한 줄만 추가):
```
hearingLossDetail HearingLossDetail?
```

추가 후 `npx prisma db push` 실행.

## TASK 2: API 엔드포인트

### 2-1. 소음성 난청 상세 CRUD
- `app/api/cases/[id]/hearing-loss/route.ts`
  - GET: HearingLossDetail + exams 전체 조회 (caseId 기준)
  - POST: HearingLossDetail 최초 생성
  - PATCH: HearingLossDetail 업데이트

- `app/api/cases/[id]/hearing-loss/exams/route.ts`
  - GET: 해당 case의 특진 목록 조회
  - POST: 특진 회차 신규 추가

- `app/api/cases/[id]/hearing-loss/exams/[examId]/route.ts`
  - PATCH: 특진 회차 업데이트
  - DELETE: 특진 회차 삭제

params는 Next.js 16 방식으로: `{ params }: { params: Promise<{ id: string }> }`
const { id } = await params

### 페이지 레이아웃

#### 상단 고정: 재해자 + 사건 기본 정보 카드
- 성명 / 생년월일 / 연락처 / 주소

- 현재 진행 단계 (Case.status 한글 표시)
- 영업 담당자 / 실무 담당자 / TF / 지사
- 메모 (인라인 편집 가능)

#### 진행 단계 스텝바 (상단 카드 아래)
아래 순서대로 스텝 표시. 현재 status에 해당하는 스텝 강조.
완료된 스텝은 체크, 현재 스텝은 파란색 강조, 미래 스텝은 회색.

스텝 순서:
1. 상담·약정
2. 기본자료 수집
3. 청구서 발송
4. 특진진찰요구서
5. 특진병원 선택
6. 특진 진행 (1차→2차→3차)
7. 전문조사
8. 통장사본
9. 결정 수령
10. 검토·종결

#### 메인 컨텐츠: 아코디언 섹션 (각 스텝 클릭 시 펼침)

### 섹션 1: 초진 정보
펼침 시 표시:
- 초진 병원명 (텍스트 입력)
- 초진일 (날짜 picker) → 재해발생일과 동일
- 초진 우측 PTA (숫자 입력, dB)
- 초진 좌측 PTA (숫자 입력, dB)
- 어음명료도 (숫자 입력, %)
- 편측 40dB 기준 통과: toggle (자동 계산: 우측 또는 좌측 ≥ 40이면 true)
- 국가장애 등록 이력: toggle
- Big3 병원 여부: toggle
- [저장] 버튼

### 섹션 2: 직업력
펼침 시 표시:
직업력 목록 (동적 추가/삭제 가능한 행 구조):
각 행:
- 회사명
- 부서/직종
- 작업내용 (그라인더, 용접 등)
- 근무기간 (시작년/월 ~ 종료년/월) - 년도+월 선택
- 소음작업 여부 toggle
- 소음수준 (dB, 선택)
- 근무시간 (텍스트)
- 출처: select (고용보험 / 소득금액증명원 / 국민연금 / 건강보험자격득실 / 경력증명서 / 기타)

하단:
- 총 소음작업 경력 자동 합산 표시
- 최종 소음작업 중단 시기 (날짜 picker)
- 직업력 특이사항 (textarea)
- [+ 행 추가] 버튼 / [저장] 버튼

### 섹션 4: 특진 진행
상단에 특진 세트 탭: [최초특진] [재특진 ▼] [재재특진 ▼]
- 재특진/재재특진 탭은 기본 숨김. [+ 재특진 추가] 버튼으로 펼침.

각 특진 세트 내부: 1차 / 2차 / 3차 서브탭

특진진찰요구서 정보 (세트 상단에 1회 표시):
- 특진진찰요구서 수령일
- 진찰기간 시작~종료
- 선택된 특진병원명
- 선택확인서 제출일

특진 회차별 데이터 입력 폼 (각 탭):
- 검사일 (날짜 picker)
- 참석 직원 (User select)
- 검사 수치 입력 (2열 그리드: 좌/우 대칭):
  - 기도청력역치 우 / 좌 (dB)
  - 골도청력역치 우 / 좌 (dB)
  - 어음명료도 우 / 좌 (%)
  - ABR 우 / 좌 (dBnHL)
  - 임피던스 우 / 좌 (A/B/C/AD/AS select)
- 검사결과 신뢰성: toggle
- 의무기록지 발급: toggle (3차 전용, 1/2차는 비활성화)
- PTA 자동 계산 표시 (기도: 500/1000/2000/4000Hz 평균 — 참고용 표시만)
- 예상 장해등급 (텍스트, 3차 종료 후 입력)
- 특이사항 (textarea)
- [저장] 버튼
