## 1. 개발 취지

전국 단위 노무법인 규모·사건 처리량에 비해 업무 방식이 중복적·비효율적.  
기존 외주 개발(TBSS 구버전)은 이해도 부족·높은 몸값으로 중단됨.  
AI 활용 바이브 코딩으로 재개발 추진. 기존 프로그램은 구상하는 시스템의 일부(사건관리)에 불과했음.

## 3. 기술 스택

| 항목 | 내용 |
|------|------|
| Frontend | Next.js (App Router), TypeScript |
| Backend | Next.js API Routes |
| DB | PostgreSQL (Prisma ORM) |
| Auth | Auth.js v5 (NextAuth) |
| PDF 생성 | pdf-lib + @pdf-lib/fontkit + NotoSansKR |
| Excel 처리 | xlsx 패키지 |
| 배포 | Railway (Node 앱 + PostgreSQL, main 브랜치 auto-deploy) |
| 레포 | github.com/jjncpla/thebosang |

> **Railway 선택 이유**: Puppeteer가 서버리스(Vercel) 환경에서 Chrome 실행 불가 → Railway(일반 Linux 서버)로 전환. Vercel 프로젝트 삭제 완료.

### 주요 기술 주의사항

- **PDF 생성**: pdf-lib + fontkit + NotoSansKR. 좌표계는 mm→px 변환, y축 반전 (pdf-lib은 좌하단 원점)
- **Next.js 15/16**: `params`는 `Promise<{...}>`로 타입 선언 후 await 필수
- **DATABASE_URL 이원화**: Railway 내부 URL (`postgres.railway.internal`) — 배포 앱용 / 외부 프록시 URL (`caboose.proxy.rlwy.net:18869`) — 로컬 개발·MCP 연결용
- **DB MCP**: `@modelcontextprotocol/server-postgres`, Windows 환경에서 `cmd /c` 래퍼 필요. `.mcp.json`은 `.gitignore`에 추가 (평문 연결 문자열 포함)
- **Prisma**: `lib/generated/` 자동생성 파일 커밋 제외
- **Git 머신 전환**: `git stash` 후 pull. 원격 버전 우선

## 4. 조직 구조 이해

### 4-1. 더보상 조직
- 본사 + 전국 지사 (울산, 부산경남, 서울북부, 경기안산, 전북익산, 경북구미, 경기의정부, 강원동해, 전남여수, 대구, 부산중부, 경기수원 등)
- 직군: 노무사 / 내근직 / 외근직

### 4-2. TF 개념
- TF = "사건군" (임시 조직 개념 아님)
- **더보상TF**: 더보상 소속 노무사·외근직이 영업한 사건 묶음 (모든 상병 취급)
- **이산TF**: 노무법인 이산 직원이 영업한 사건 묶음 (소음성 난청 전용)
- 각 지사가 담당하는 TF가 고정되어 있음 (예: 울산지사 → 더보상울산TF, 이산울산동부TF, 이산울산남부TF, 이산울산북부TF)
- TF명 정규화 적용 (예: "울동" → "울산동부TF")
- **담당 TF 배정표 전달 완료** → 사건목록 페이지 관할표에 구현됨. 이 관할표가 기준.

### 4-3. 노무법인 이산
- 더보상의 협력 법인. 이산 소속 직원이 영업한 사건을 더보상이 수행
- 이산 임직원은 TBSS에서 **사건 조회 페이지만** 제한적으로 접근 가능
- 상담관리 TF 범위: **더보상TF만** (이산TF 제외)

## 7. 페이지별 상세 기획

### 7-1. 로그인 페이지

- 레이아웃:
  - 상단: [대표님 인사말] (좌측 박스) + [관리자 공지사항] (우측 박스)
  - 하단: 로그인 입력창 (가로로 넓게)

- 로그인 성공 시: To Do List 페이지로 이동
- 대표 인사말 / 공지사항: 관리자 페이지에서 작성·수정

---

### 7-2. To Do List (로그인 후 첫 화면)

- 개념: 로그인한 사용자가 담당자로 지정된 업무들을 날짜별로 모아서 보여주는 페이지
- 레이아웃:
  - 좌측 1/2: 달력 + 날짜별 To Do 이력
  - 우측 1/2 상단: 요청 유형별 필터링 (정산 요청, 이의제기 작성 요청, 제척 임박 등)
  - 우측 1/2 하단: 기간 설정 조회
  - 하단 여백: 자유 기술 메모 공간
- 연동: 이의제기 기일 관리에서 제척 도래 시 자동 반영

### 7-3. 사건 관리 (핵심 기능)

#### 개념
- 병원 환자 관리 프로그램 컨셉 차용
- 환자(재해자) 고정정보 기준으로 사건 조회·관리
- 사건 단계: 상담(약정 전) → 진행 중 → 종료 → 파생 사건 관리

#### 사건 분류 (2개 계열)
- **기획사건**: 소음성 난청, COPD, 진폐증 — 초진 → 특진/정밀 → 전문조사 → 장해급여 결정 흐름
- **일반산재**: 근골격계, 업무상 사고, 직업성 암, 유족 등 — 접수 → 전문조사/질판위 → 요양승인 → 요양 → 장해청구 흐름

#### 데이터 구조
- **공통 DB** (재해자 고정정보): 성명, 주민번호, 연락처, 주소 등
- **사건별 DB**: 상병별 확장 테이블 (섹션 14 PRD 참조)
- 재해자 1명 = 복수 상병 = 복수 사건 가능
- 파생 사건 (평균임금 정정 청구 등): `parentCaseId` FK로 원본 연결
- 통합 상세 페이지: `/patients/[id]` (재해자 단위 진입점)

#### 핵심 아키텍처: B방식 FK 연결
```
Patient (재해자)
  └─ Case (사건) ─────────────────────────────────┐
       └─ HearingLossDetail / COPDDetail / ...     │
            └─ HearingLossExam (특진 회차별)        │
                                                   │
ObjectionReview (처분검토) ─ caseId FK ────────────┘
  └─ ObjectionCase (이의제기/소송인계/평임정정)
       └─ parentCaseId → Case (파생 사건 연결)
```

#### 주요 기능
- 사건 목록 조회 + 다중 필터링 (상병, 진행단계, 담당자, TF, 지사 등)
- 재해자목록: TF 필터 포함
- 진행상황 한글화 표시
- 사건 상세: 진행경과 기록, 메모, 관련 문서 첨부
- 보험급여 청구서식 자동 생성 (PDF) → 사건 상세 하위 기능으로 통합

## 14. 사건관리 PRD

> 스키마 설계 및 UI 구현 참조 문서. Phase 2에서 DB 연계 및 스키마 확정 완료.

---

### 14-0. 핵심 아키텍처: B방식 FK 연결

```
Patient (재해자)
  └─ Case (사건)
       ├─ HearingLossDetail (소음성 난청 상세)
       │    └─ HearingLossExam (특진 회차별, examSet + examRound)
       ├─ COPDDetail (추후)
       └─ PneumoconiosisDetail (추후)

ObjectionReview (처분검토)
  └─ caseId: String → Case.id FK  ← B방식 핵심
  └─ ObjectionCase (이의제기 / 소송인계 / 평균임금정정)
       └─ parentCaseId → Case.id (파생 사건 연결)
```

> **B방식 원칙**: ObjectionReview는 Case를 참조하는 FK를 보유. Case가 ObjectionReview를 모름. 단방향 참조. 현재 스키마에 적용 완료.

---

### 14-1. 공통 구조

#### 핵심 설계 원칙
- **Patient(재해자)** 가 최상위 엔티티. 1명이 복수 상병(사건)을 가질 수 있음
- **Case(사건)** 은 항상 특정 상병에 귀속. 상병이 다르면 별개 사건
- **상병 계열** 구분:
  - 기획사건: 소음성 난청 / COPD / 진폐증
  - 일반산재: 근골격계 / 업무상 사고 / 직업성 암 / 유족 / 기타
- **파생 사건**: 평균임금 정정 청구 시 원본 `caseId`를 `parentCaseId`로 연결하여 새 사건 생성

#### 공통 상태값 (CaseStatus Enum — 기획사건용)
```
CONSULTING              상담 중 (초진 전, 약정 전)
CONTRACTED              약정 완료 / 진행 개시
DOC_COLLECTING          기본자료 수집 중
SUBMITTED               공단 접수 완료 (우편 발송)
EXAM_REQUESTED          특진진찰요구서 수령
EXAM_CLINIC_SELECTED    특진병원 선택확인서 제출 완료
EXAM_SCHEDULED          특진 일정 확정
IN_EXAM                 특진 진행 중
EXAM_DONE               특진 완료 (3차 종료 + 의무기록지 수령)
EXPERT_REQUESTED        전문조사진찰요구서 수령
EXPERT_CLINIC_SELECTED  전문조사기관 선택 완료
EXPERT_DONE             전문조사 완료
BANK_REQUESTED          통장사본 요청 수령 (전화)
BANK_SUBMITTED          통장사본 제출 완료
DECISION_RECEIVED       장해등급결정통지서 수령
REVIEWING               노무사 검토 중 (평정/적사변 실익)
INFO_REQUESTED          정보공개청구 중
APPROVED                승인 확정
REJECTED                불승인
CLOSED                  종결
OBJECTION               이의제기 진행 중 (이의제기 관리로 이관)
WAGE_CORRECTION         평균임금 정정 청구 (파생 사건 생성)
```

#### Case 공통 필드
- `caseType`: Enum (HEARING_LOSS / COPD / PNEUMOCONIOSIS / MUSCULOSKELETAL / OCCUPATIONAL_ACCIDENT / OCCUPATIONAL_CANCER / BEREAVED / OTHER)
- `parentCaseId`: String? — 파생 사건 연결 (평균임금 정정 청구 등)
- 담당자 필드: `salesManagerId`, `caseManagerId`, `branchManagerId` — User FK

---

### 14-2. 소음성 난청 PRD

#### 절차 흐름
```
CONSULTING
  └─ 초진 통과 (편측 PTA 40dB↑) + 약정서·위임장 수령 → CONTRACTED
  └─ 기본자료 수집 시작 → DOC_COLLECTING
  └─ 청구서식 작성 + 우편 발송 → SUBMITTED
  └─ 특진진찰요구서 수령 (웹팩스) → EXAM_REQUESTED
  └─ 특진의료기관 선택확인서 제출 → EXAM_CLINIC_SELECTED
  └─ 특진 일정 확정 → EXAM_SCHEDULED
  └─ 1차 특진 → IN_EXAM (examRound=1)
  └─ 2차 특진 → IN_EXAM (examRound=2)
  └─ 3차 특진 + 의무기록지 발급 → EXAM_DONE
  └─ 전문조사진찰요구서 수령 → EXPERT_REQUESTED
  └─ 전문조사기관 선택확인서 제출 → EXPERT_CLINIC_SELECTED
  └─ 전문조사 실시 → EXPERT_DONE
  └─ 통장사본 요청 수령 → BANK_REQUESTED
  └─ 통장사본 팩스 제출 → BANK_SUBMITTED
  └─ 장해등급결정통지서 수령 → DECISION_RECEIVED
  └─ 노무사 평정/적사변 실익 검토 → REVIEWING
       ├─ 정공 필요 → INFO_REQUESTED → 자료 수령 후 재검토
       ├─ 이상 없음 → CLOSED
       ├─ 평균임금 정정 → WAGE_CORRECTION (파생 사건)
       └─ 불승인 → REJECTED
            ├─ 종결 → CLOSED
            └─ 이의제기 → OBJECTION
```
